# RFC 006：用户可见配置与公开契约演进策略

## 状态

Draft

## 日期

2026-06-20

## 背景

当前仓库里已经存在多类用户可见契约：

- `package.json` 中的 `contributes.configuration`
- 主题相关公开配置项
- 已起草的主题命令面
- README 中对“已支持能力”的公开承诺
- RFC 中声明为稳定的结构、class、`data-*` 与行为语义

这些内容一旦被用户依赖，就不再只是内部实现细节，而是项目的公开契约。

如果没有一份明确的演进策略，后续很容易出现这些问题：

- 代码先变，文档后补，最终描述与行为不一致
- 配置项、命令名或稳定结构被随意改动，破坏已有使用方式
- 新能力进来时，没有判断它到底是内部实现、公开行为，还是新的契约扩张
- README、RFC、验证基线之间的承诺边界逐渐漂移

这份 RFC 的目标，就是明确：哪些东西算公开契约、如何修改、修改时必须同步做什么。

## 目标

定义：

- 公开契约的范围
- 契约变更的分类
- 契约变更时的必需动作
- 文档、实现与验证的一致性要求

## 非目标

这份 RFC 不负责：

- 定义具体主题配置模型
- 定义具体命令交互方式
- 定义具体插件结构
- 定义版本号策略细节

这份 RFC 只关注“公开契约如何演进”。

## 决策

### 1. 下面这些内容一律视为公开契约

当前项目中，以下内容属于公开契约：

1. 用户可见配置项
2. 用户可见命令
3. README 中声明为“已支持”的能力
4. RFC 中声明为稳定的结构与行为
5. `package.json` 中对外暴露的扩展贡献点行为

在当前仓库里，至少包括：

- `vscode-github-markdown.theme.mode`
- `vscode-github-markdown.theme.single`
- `vscode-github-markdown.theme.system.day`
- `vscode-github-markdown.theme.system.night`
- `vscode-github-markdown.changeThemeMode`
- `vscode-github-markdown.changeThemeSingle`
- `vscode-github-markdown.changeThemeSystemDay`
- `vscode-github-markdown.changeThemeSystemNight`
- `.vscode-github-markdown`
- `data-color-mode`
- `data-light-theme`
- `data-dark-theme`
- README 中列出的已支持能力：
  - 任务列表
  - 脚注
  - Alerts
  - Emoji 短代码
  - Mermaid 图表 CSS 样式
  - GitHub 主题系统（9 套主题、两种模式）
  - 通过 VS Code 命令切换主题

这些内容不能再被当作“随便改的内部实现”处理。

### 2. 契约变更分为三类

所有公开契约变更统一分为三类：

1. 新增
2. 收紧或调整
3. 移除或重命名

#### 新增

示例：

- 新增一个公开配置项
- 新增一个公开命令
- 新增一个被 README 正式声明支持的能力

要求：

- 必须先定义语义
- 必须补文档
- 必须补验证

#### 收紧或调整

示例：

- 调整一个配置项的合法值集合
- 调整某个稳定 `data-*` 的语义
- 调整 README 中对支持范围的描述

要求：

- 必须明确旧行为和新行为的差异
- 必须判断是否会破坏已有使用者预期
- 必须同步更新 RFC、README 与验证

#### 移除或重命名

示例：

- 删除配置项
- 重命名命令
- 删除已承诺的稳定结构

要求：

- 默认视为高风险变更
- 不应直接发生，除非有非常明确的理由
- 必须先经过专门 RFC 或同等级别说明

### 3. 公开契约默认保守演进

公开契约的默认策略是保守，而不是灵活。

这意味着：

- 先新增，再考虑替换
- 能保持兼容，就不要破坏兼容
- 能作为内部实现改动解决，就不要扩大到公开契约层

特别是以下内容应保持更高稳定性：

- 配置 key
- 命令 id
- 根容器 class
- 预览层依赖的稳定 `data-*`

### 4. 没有文档和验证的功能，不得宣称为公开支持

一个能力只有同时满足以下条件，才能进入公开承诺：

1. 实现已经存在
2. README 或等价用户文档已说明
3. 有最小验证方式

缺一不可。

这条规则尤其适用于：

- 新增 GitHub 兼容性能力
- 新增主题相关命令
- 新增稳定结构依赖

### 5. README、RFC、实现、验证必须四方一致

当公开契约发生变化时，至少要检查四个地方：

1. 实现
2. RFC
3. README / 用户文档
4. 验证

目标不是“代码改完就算完”，而是：

- 实现说它支持什么
- 文档也说它支持什么
- RFC 解释为什么这样设计
- 验证能证明这件事真的成立

如果四者不一致：

- 以已验证、已文档化、已正式声明的部分为准
- 不应让 README 抢跑实现

### 6. 契约外的内容默认视为内部实现

以下内容默认属于内部实现，可在不破坏契约的前提下自由调整：

- 插件内部局部算法
- token 处理中未对外声明的细节
- 样式实现中的非稳定中间结构
- 命令内部如何组织选择逻辑

但前提是：

- 不破坏稳定输出
- 不破坏用户配置语义
- 不破坏已验证行为

### 7. README 的“已支持能力”是高优先级承诺

README 中写出的“当前已覆盖的语法”属于高优先级公开承诺。

因此：

- 新增到这份列表的能力，必须先有实现与验证
- 从这份列表移除能力，必须明确说明原因
- 不应把“计划支持”写成“已经支持”

当前这条规则尤其重要，因为项目目标本身就高度依赖“真实能力边界”的诚实表达。

### 8. 配置与命令变更必须优先考虑兼容迁移

如果未来确实需要调整：

- 配置 key
- 命令 id
- 主题值命名

则优先级应为：

1. 保持原契约不变
2. 如果必须变，先提供兼容路径
3. 如果连兼容路径都做不到，必须先写专门说明

也就是说，不应因为内部实现更顺手，就直接推翻既有公开命名。

## 公开契约变更检查表

当某个改动触及公开契约时，至少要回答：

- 这次改动触及的是否是公开契约
- 它属于新增、调整，还是移除
- README 是否需要同步修改
- 对应 RFC 是否需要同步修改
- 对应验证是否需要补或改
- 是否会破坏已有用户配置或已有使用方式

如果这些问题答不清楚，就不应把改动当作普通内部重构处理。

## 对当前实现的解释

基于这份 RFC，当前仓库可以理解为：

- 主题配置项已经属于公开契约
- 主题命令一旦正式贡献，也会立即成为公开契约
- `README.md` 和 `README.zh-CN.md` 中的支持列表不能超前于实现
- `RFC 002` 和 `RFC 003` 中声明的稳定结构与语义，不应在无说明的情况下被破坏

## 对后续工作的影响

这份 RFC 生效后：

- 未来任何新增配置、命令或稳定结构，都应先判断是否进入公开契约
- 兼容性相关 PR 不能只改代码，不改文档与验证
- 如果要调整主题配置模型或命令名，应优先走 RFC 更新
- 发布前检查应包含“公开契约一致性”这一项

## 开放问题

- 是否需要单独维护一份公开契约清单文档
- 如果未来引入 deprecation 机制，是否需要单独写一篇 RFC
- 是否要把 README 的“已支持能力”自动对照到验证基线
