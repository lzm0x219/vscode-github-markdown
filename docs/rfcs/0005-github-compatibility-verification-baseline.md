# RFC 005：GitHub 兼容性验证基线

## 状态

Draft

## 日期

2026-06-20

## 背景

当前路线图已经明确要求：

- 已支持能力需要可重复验证
- GitHub 兼容性不应只停留在“看起来差不多”
- 后续新增能力不能轻易破坏已支持行为

仓库里已经有一条很小但真实存在的验证基础：

- `scripts/verify-github-markdown.ts`（需要 [Bun](https://bun.sh) 运行时）

它当前覆盖了三项已支持能力：

- 任务列表
- 脚注
- Alerts

但目前还缺少一份正式的验证基线文档，来回答这些问题：

- 什么才算“最小但有效”的兼容性验证
- 哪些能力必须进入固定验证集
- 什么差异算可接受，什么差异必须修复
- 哪些检查属于开发时回归检查，哪些属于发布前门槛

这份 RFC 的目标，就是把验证范围、验证层次和最小门槛固定下来。

## 目标

定义：

- GitHub 兼容性验证的最小固定集合
- 各类能力对应的验证方式
- 可接受差异与不可接受差异的判断口径
- 开发时检查与发布前检查的分层

## 非目标

这份 RFC 不负责：

- 定义所有未来能力的完整测试矩阵
- 定义浏览器自动化或截图对比的具体实现细节
- 定义主题配置模型
- 定义主题命令面

这些内容应由其他 RFC 或后续实现细化。

## 决策

### 1. 验证分为三层

GitHub 兼容性验证统一分为三层：

1. 插件输出验证
2. 预览结构验证
3. 人工对照验证

#### 插件输出验证

目标：

- 验证 `markdown-it` 插件输出是否符合预期
- 覆盖最容易被回归破坏的 HTML 结构、class 和 attribute

特点：

- 快
- 可重复
- 适合日常开发回归

当前基础：

- `scripts/verify-github-markdown.ts`

#### 预览结构验证

目标：

- 验证主题容器、主题元数据、预览层依赖的稳定结构没有被破坏
- 验证扩展层和插件层之间的契约仍然成立

特点：

- 比纯插件验证更接近真实运行链路
- 不一定需要完整视觉比对

#### 人工对照验证

目标：

- 验证“接近 GitHub”的最终体验
- 检查视觉细节、留白、可读性和整体观感

特点：

- 慢
- 主观成分更高
- 但在当前阶段仍然必要

### 2. 最小固定验证集必须覆盖已实现能力

最小固定验证集必须至少覆盖当前已实现的 `extension-owned` 能力：

- 任务列表
- 脚注
- Alerts

以及应尽快补入的高价值兼容性候选：

- 表格视觉对齐
- 代码块视觉对齐
- 标题锚点行为

也就是说：

- 已实现能力必须进入固定验证集
- 已列入路线图重点的兼容性候选，应尽快进入验证集

### 3. 插件输出验证以 fixture + 断言为基线

对于语法增强插件，最小验证方式固定为：

- 输入 Markdown fixture
- 执行渲染
- 对输出 HTML 进行关键断言

当前已存在可复用模式：

- 任务列表断言 `contains-task-list`
- 断言 checkbox 结构与 `checked disabled`
- 脚注断言 `.footnote-ref`、`.footnotes`
- Alerts 断言 `.markdown-alert-*` 和标题内容

后续新增插件验证时，应优先沿用这种模式，而不是先上更重的测试系统。

### 4. 预览结构验证必须覆盖稳定契约

凡是在 `RFC 002` 和 `RFC 003` 中被声明为稳定结构的内容，都应进入结构验证范围。

至少包括：

- 根容器 `.vscode-github-markdown`
- `data-color-mode`
- `data-light-theme`
- `data-dark-theme`
- 任务列表相关稳定 class
- Alerts 相关稳定 class
- 脚注相关结构与 `data-*`

如果这些稳定结构发生变化：

- 要么视为回归
- 要么先修改对应 RFC 和文档，再变更实现

### 5. 人工对照验证只检查高价值视觉差异

人工对照验证不追求面面俱到，而是集中在高价值差异：

- 标题层级与留白
- 段落间距
- 列表缩进与勾选框可读性
- 引用块与 Alerts 观感
- 代码块边框、间距与整体可读性
- 表格边框、对齐与内边距
- 脚注链接、回跳与尾部区域可读性

人工对照的标准不是“像某个主题”，而是：

- 是否明显接近 GitHub
- 是否违背 README 和 ROADMAP 的公开承诺
- 是否出现误导性的伪兼容

### 6. 可接受差异与不可接受差异

可接受差异：

- 与 GitHub 存在少量视觉细节偏差，但不影响语义和整体观感
- 由于 VS Code 预览宿主限制导致的局部非关键差异
- 未承诺支持能力上的自然退化

不可接受差异：

- 已声明支持的能力出现语义错误
- 稳定 class 或 `data-*` 丢失，导致预览层失效
- 结构或链接行为错误，例如脚注引用断裂
- 输出具有误导性，看起来像支持，实际行为不正确

### 7. 检查分为开发时门槛和发布前门槛

#### 开发时门槛

每次改动兼容性相关实现时，至少应运行：

- 对应的最小插件输出验证
- 受影响能力的结构验证

对于当前仓库，最小起点是：

- `bun scripts/verify-github-markdown.ts`

#### 发布前门槛

发布前应至少完成：

- 所有固定验证集通过
- 关键人工对照检查完成
- 文档中“已支持能力”与实际验证范围一致

也就是说，发布前门槛高于开发时门槛，但不应重到阻碍日常迭代。

### 8. 新能力进入路线图时，也必须进入验证计划

后续新增 GitHub 兼容性能力时：

1. 先定义它属于哪一类能力
2. 再定义它的最小验证方式
3. 没有最小验证方式的能力，不应宣称为已支持

这条规则适用于：

- 新插件
- 新主题行为
- 新结构性兼容补齐

## 推荐最小验证清单

当前推荐的最小固定验证清单如下：

| 能力                   | 验证层级            | 当前状态 | 说明                                              |
| ---------------------- | ------------------- | -------- | ------------------------------------------------- |
| 任务列表               | 插件输出验证        | Exists   | 已有脚本覆盖，后续可继续扩 fixture。              |
| 脚注                   | 插件输出验证        | Exists   | 已有脚本覆盖，后续可补更多边界样例。              |
| Alerts                 | 插件输出验证        | Exists   | 已有脚本覆盖。                                    |
| 主题根容器与主题元数据 | 预览结构验证        | Planned  | 应补 `.vscode-github-markdown` 与 `data-*` 断言。 |
| 表格视觉对齐           | 人工对照验证        | Planned  | 当前仍需人工对照结论。                            |
| 代码块视觉对齐         | 人工对照验证        | Planned  | 当前仍需人工对照结论。                            |
| 标题锚点行为           | 结构验证 + 人工对照 | Planned  | 需先明确目标行为。                                |

## 对当前实现的解释

基于这份 RFC，当前仓库可以理解为：

- `scripts/verify-github-markdown.ts` 是最小验证基线的第一步
- 当前已支持能力里，任务列表、脚注、Alerts 已经有基础断言
- 主题结构、表格、代码块、锚点还需要继续补验证覆盖

## 对后续工作的影响

这份 RFC 生效后：

- 后续实现兼容性能力时，必须同时补对应验证
- README 中“已支持能力”的范围，必须与已验证范围保持一致
- 主题相关 RFC 和插件契约 RFC 中声明的稳定结构，应在这里获得验证落点
- 发布前检查不应只依赖构建和 lint

## 开放问题

- 是否需要为验证专门建立 fixtures 目录
- 预览结构验证是否要引入更贴近 VS Code 运行态的脚本
- 人工对照验证是否需要固定检查清单文档
