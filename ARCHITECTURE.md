# 架构设计

## 目标

`vscode-github-markdown` 是一个轻量级 VS Code 扩展，目标是在离线、本地编辑场景下，让 Markdown 预览尽可能接近 GitHub 的渲染行为与呈现效果。

本项目的架构刻意保持收敛：

- 复用 VS Code 内置 Markdown 预览流水线
- 在 GitHub 存在确定性语法差异的地方扩展 `markdown-it`
- 在预览层补齐 GitHub 风格的展示效果
- 明确不重建独立的 Markdown 预览系统

## 架构原则

### 1. 基于平台扩展，而不是替换平台

VS Code 已经负责文档加载、Markdown 预览生命周期、Webview 宿主以及基础 Markdown 渲染。本项目只通过官方贡献点插入必要的扩展行为。

### 2. 只处理离线可确定的 GitHub 一致性

扩展只实现那些能够从 Markdown 源文本或本地配置中推导出的行为。任何依赖 GitHub 仓库状态、网络请求或服务端上下文的能力都不属于目标范围。

### 3. 语法增强保持小而独立

每一个 GitHub 特有的语法差异都应该落在一个聚焦的 `markdown-it` 插件或一个局部预览规则里，而不是堆进一个庞大的渲染框架。

### 4. 运行时保持 Web Extension 兼容

当前清单同时声明了 `main` 和 `browser`，因此运行时代码必须保持 Web Extension 可用。除非项目明确调整目标，否则架构不应依赖 Node 专属 API。

### 5. 呈现效果必须局部作用

所有预览侧样式和 DOM 假设都必须限制在扩展自己的容器内，不能污染 VS Code 预览宿主的其它区域。

## 系统定位

这个扩展位于 VS Code Markdown 预览基础设施与最终渲染结果之间，承担“补齐 GitHub 差异”的职责。

```mermaid
flowchart LR
  A["Markdown 文档"] --> B["VS Code Markdown Preview"]
  B --> C["扩展激活与 markdown-it 接入"]
  C --> D["GitHub 一致性插件链"]
  D --> E["带主题元数据的 HTML 包装"]
  E --> F["预览样式与预览脚本"]
  F --> G["VS Code 中的最终预览结果"]
```

## 分层架构

### 1. Manifest 与贡献层

**核心文件：** `package.json`

这一层负责声明扩展如何接入 VS Code：

- 扩展入口声明：`main`、`browser`
- Markdown 预览贡献点：
  - `markdown.markdownItPlugins`
  - `markdown.previewScripts`
  - `markdown.previewStyles`
- 用户配置模型
- 激活条件

这一层是声明式层，不承载业务逻辑。它负责定义集成契约，而不是实现 Markdown 处理细节。

### 2. 扩展宿主层

**核心文件：** `src/extension.ts`、`src/events.ts`、`src/theme.ts`

这一层负责扩展激活期和运行期的轻量协调：

- 通过 `extendMarkdownIt` 注册扩展的插件链
- 监听与配置变更相关的刷新事件
- 读取并归一化主题相关配置
- 为渲染相关逻辑提供稳定的配置访问入口

这一层应尽量薄，只做编排，不直接承担 Markdown 解析职责。

### 3. Markdown 转换层

**核心目录：** `src/plugins/*`

这一层是项目的核心行为层，用于处理 GitHub 特有且可离线确定的 Markdown 语法差异。

职责包括：

- 检查 `markdown-it` token 流
- 以最小必要改动修正 GitHub 相关语义或输出结构
- 在无法可靠匹配时保留默认 Markdown 行为
- 让每个特性按职责拆分，彼此独立

这一层的插件应满足以下约束：

- 一个插件只负责一个特性族
- 不依赖网络访问
- 不假设仓库在线上下文
- 不引入隐式跨插件耦合
- 输出足够稳定的 HTML 结构或 CSS hook，供预览层消费

任务列表、Alerts、Emoji 短代码、脚注、主题容器注入，都属于这一层的典型职责。

`markdown-it-github-image-url` 属于渲染修正：在所有语义和主题处理完成后，将图片路径从绝对路径（`/path`）重写为相对路径（`./path`），使项目内绝对路径图片可在 webview 中正常加载。

### 4. 预览呈现层

**核心文件：** `src/extension.preview.css`、`src/extension.preview.ts`

这一层负责 Markdown 被转换后的最终展示：

- GitHub 风格的排版、留白、颜色与组件样式
- Light / Dark / System 模式下的主题适配
- 仅在 CSS 不足以表达时才增加少量预览侧 DOM 行为

这一层必须始终建立在扩展生成的根容器之上，避免样式泄漏到宿主预览环境之外。

## 运行流程

系统整体应保持单向、简单、可推导：

1. VS Code 为某个 Markdown 文档打开预览。
2. 扩展因 Markdown 使用场景被激活。
3. `extendMarkdownIt` 注册扩展维护的插件链。
4. VS Code 使用 `markdown-it` 执行渲染。
5. 各插件对 token 流或渲染输出做 GitHub 一致性修正。
6. 最终 HTML 被包裹进扩展自有的根容器，并附带主题元数据。
7. 预览样式和可选脚本完成最终呈现。
8. 当相关配置变化时，扩展触发预览刷新。

目标架构中不应出现以下机制：

- 自建渲染服务
- 外部请求驱动的预览增强
- 常驻后台同步循环

## 层间稳定契约

### Manifest -> 扩展宿主层

- `package.json` 声明贡献点与配置模型
- 运行时代码必须把这些声明视为公开契约
- 配置项属于用户可见 API，应谨慎演进

### 扩展宿主层 -> Markdown 转换层

- 宿主层决定插件注册顺序
- 插件应在单次渲染内保持可预测、近似纯函数式的行为
- 插件可以读取配置辅助函数，但不应依赖除配置之外的编辑器全局可变状态

### Markdown 转换层 -> 预览呈现层

- 转换层可以输出 GitHub 一致性所需的结构化 HTML 与样式 hook
- 预览层只能依赖扩展自有的 class、data attribute 与稳定结构
- 如果某个特性必须依赖脆弱 DOM 假设，优先简化或放弃，而不是把复杂度转嫁给预览层

## 配置模型

当前架构只保留一个很小的配置面，核心围绕主题选择：

- 模式配置决定使用单一主题还是跟随系统昼夜模式
- 主题值在渲染时被归一化为根容器上的数据属性
- CSS 依据这些属性决定最终展示

配置是系统中最主要的可变状态。架构上刻意不引入额外状态子系统，例如缓存、后台索引或工作区级特性注册表。

## 支持范围模型

项目中的 Markdown 行为应分为三类：

### 1. VS Code 已经处理得足够好的能力

直接复用内置预览，不重复实现。

### 2. GitHub 特有但本地可确定复现的能力

使用小型插件和局部样式补齐。

### 3. 依赖在线或仓库上下文的 GitHub 能力

明确标记为不支持，并在文档中说明，而不是做不可靠模拟。

这是避免项目演变成“在编辑器里重建 GitHub”的首要护栏。

## 非目标

本架构明确不追求以下方向：

- 脱离 VS Code 预览体系的独立 Markdown 渲染器
- GitHub API 集成
- 仓库上下文感知的引用解析
- 与 GitHub 一致性无关的额外主题系统
- 只有单一用途却被抽象成通用框架的大型层次结构

## 失败语义

当扩展无法可靠复现某个 GitHub 特性时，优先遵守以下退化策略：

- 保留标准 Markdown 预览行为
- 不输出具有误导性的 HTML
- 不做只在简单场景看起来正确的半成品模拟

也就是说，保守地“不命中”比自信地“伪正确”更可取。

## 演进策略

后续新增能力应按以下顺序判断：

1. 先确认 VS Code 内置预览是否已经覆盖
2. 如果没有，再判断该能力是否可以离线确定
3. 如果可以，用最小插件或局部样式补齐
4. 如果不可以，明确记录为不支持

这样可以保证项目始终围绕真正目标演进：提高可靠的 GitHub Markdown 一致性，而不是追求无边界的功能扩张。
